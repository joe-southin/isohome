# Personal Website Stack Profile

This file describes the target architecture for Joe's personal website. Reference it
whenever a project may be deployed to or integrated with the site — and use it to make
spec decisions that align with the existing stack rather than introducing new patterns.

---

## Platform overview

A Cloudflare-native SPA + edge API:

```
Browser
  └── Static SPA (Vite build, served via ASSETS binding)
        └── Fetches from CF Worker API (/api/*)
              ├── D1 (SQLite database at the edge)
              ├── R2 (object/file storage)
              └── Cloudflare Images (image processing/serving)
```

No traditional server. No Node.js runtime. All backend logic is a Cloudflare Worker
(JS/TS, edge runtime). The SPA and Worker share a single Cloudflare deployment.

---

## Frontend

**Language**: TypeScript
**Build tool**: Vite + SWC
**Routing**: React Router v6
**Server state**: TanStack Query v5
**UI components**: shadcn/ui (Radix UI primitives + Tailwind CSS)
**Forms**: React Hook Form + Zod
**Charts**: Recharts
**Content**: react-markdown + KaTeX

### How this shapes spec decisions

**UI components** — before speccing a custom component, check whether a shadcn/ui
primitive covers it. shadcn/ui components are preferred. Custom components should be
Tailwind-only (no separate CSS files, no CSS-in-JS).

**Data fetching** — server state is managed via TanStack Query. Spec'd API calls should
be shaped as `useQuery` / `useMutation` hooks, not raw `fetch` calls. Include the
suggested query key structure when speccing new endpoints.

**Forms** — all forms use React Hook Form with a Zod schema for validation. When
speccing a form, define the Zod schema explicitly. The schema doubles as the API
request validation on the Worker side.

**Routes** — new pages get a React Router v6 route entry. Spec'd routes should note the
URL pattern, any loader data requirements, and whether they need auth protection.

## Backend: Cloudflare Worker

**Language**: JavaScript (worker.js entry point), TypeScript types via JSDoc or `.d.ts`
**Runtime**: Cloudflare Workers edge runtime — not Node.js. No `fs`, `net`, `path`,
`child_process`, or other Node built-ins. Web-standard APIs only (`fetch`, `URL`,
`crypto`, `ReadableStream`, etc.).

**Request routing pattern**: Workers handle `fetch` events. Pattern looks like:

```js
// worker.js
export default {
  async fetch(request, env, ctx) {
    const url = new URL(request.url);
    if (url.pathname.startsWith('/api/')) {
      return handleApi(request, env, url);
    }
    return env.ASSETS.fetch(request); // fall through to SPA
  }
};
```

### How this shapes spec decisions

When speccing API endpoints, define them as Worker handlers. For each endpoint specify:
- Method + path (e.g., `GET /api/books/:id`)
- Request body shape (JSON, typed)
- Response body shape (JSON, typed)
- Auth requirement (Bearer token checked in Worker)
- Which binding it uses (D1, R2, or Images)

No Express, Hono, or other HTTP framework is assumed — but if a router lib is added,
note it in the spec.

## Database: Cloudflare D1 (SQLite at the edge)

D1 is SQLite. All schema decisions must be SQLite-compatible. Key constraints:
- No `ARRAY`, `JSON` column types — store as TEXT and parse in the Worker
- No `ENUM` types — use CHECK constraints or a lookup table
- Auto-increment primary keys: `INTEGER PRIMARY KEY` (not `SERIAL`)
- Timestamps: store as `TEXT` in ISO 8601 format, or `INTEGER` as Unix ms

### How this shapes spec decisions

When speccing a data model that will be persisted, include the CREATE TABLE DDL.
Example:

```sql
CREATE TABLE books (
  id         INTEGER PRIMARY KEY,
  title      TEXT NOT NULL,
  author     TEXT,
  finished   INTEGER NOT NULL DEFAULT 0,  -- boolean as 0/1
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

Also spec the D1 query patterns used by the Worker (not just the schema).

---

## Object storage: Cloudflare R2

R2 is S3-compatible object storage. Use for:
- User-uploaded files
- Static assets too large or dynamic for the SPA bundle
- Export files (CSV, PDF) generated by the Worker

When speccing a feature that involves file storage, note:
- The R2 bucket binding name (e.g., `env.BUCKET`)
- The object key pattern (e.g., `uploads/{userId}/{filename}`)
- Whether objects are public (served via R2 public URL) or private (signed URL or
  proxied through the Worker)

---

## Cloudflare Images

Used for image upload, transformation, and serving. When speccing image handling, note
whether images need resizing/transformation at serve time (use CF Images) versus static
serving (use R2 direct).

---

## Common patterns to follow in specs

**Auth** — JWT-based. Token validated in the Worker on protected routes. Include auth
requirement on each API endpoint in the spec.

**Error handling** — Workers return JSON error responses: `{ error: string, code?: string }`.
Spec error cases explicitly for each endpoint.

**Environment / secrets** — Worker env bindings (D1, R2, secrets) are accessed via `env`
parameter. Never hardcode credentials. When a new secret is needed, note it in the spec
as an `env` binding.

**TypeScript types** — define shared types (request/response shapes, D1 row types) in a
`types.ts` or similar. Spec'd data models should translate directly into TypeScript
interfaces.

---

## When a project is NOT for the personal website

If a project is standalone (a CLI, a local script, a separate service), this stack
profile doesn't apply. Don't force Cloudflare or React patterns onto something that
doesn't need them. Use this profile only when the integration path is explicit.
